数据传输协议

数据传输协议

version 3.0

数据流向

协议的用户由闸机、后台和控制端三种身份的用户组成。

在系统投入使用前，控制端应向后台发送“注册闸机”和“注册游客id”的指令。后台只会处理已注册闸机的请求，同时只认定已注册的游客id为合法。后台会输出指令的执行结果。

在系统正式使用时，闸机取得游客的体温和存放在他芯片上的id，并请求后台判断id是否合法、体温是否正常。后台返回判断结果。

综上，数据有四种流向：

  • 闸机向后台发送请求（rqst）
  • 后台向闸机回复请求（rply）
  • 控制端向后台发送命令（cmd）
  • 后台向控制端返回输出（cmdret）

每种流向都有对应的主题供相关方订阅和发布。

MQTT协议设置

服务质量（QoS）

绝大部分通信依赖“正好一次”级别的服务质量。即QoS=2。

消息负载（payload）格式

使用JSON格式。每个主题都规定了消息负载的格式，并给出了示例。

ID

如无特殊说明，所有种类的id不限制长度，但只能由字母或数字组成。

闸机id

每台闸机都有自己的唯一标识符（machine_id）。只有登记过的闸机才能与后台交互。

游客id

游客的id存放在其随身携带的超高频芯片中。只有登记过的游客id才合法。

指令id

控制端向后台发送指令时，会包含一个id。后台返回指令结果时，也会附带同样的id。以此来区分结果属于哪条指令。

MQTT Broker

以下是连接到MQTT Broker服务器的必要信息。此账号由闸机、后台和控制端共用。

IP：124.70.155.130
端口：1883
用户名：user1
密码：helloworld

后台错误代码

当后台返回非零的错误代码时，代表后台发生了异常。此时，与错误代码一同返回的其他所有值均无效。错误代码的含义如下表所示：

代码 错误名 含义
0 无异常 无错误发生，其他值有效。
1 服务器异常 由于服务器自身原因无法处理请求。
2 不合法的消息 无法解析提供的JSON消息。
